# AIBridge Repository Guidelines

## 目标
本仓库以 `AIBridge` 为基础，维护一个面向 Unity 的轻量 AI 桥接层。

- `main`：功能演进主线。允许新增能力、重构、工具链升级和实验性模块。
- `unity2021`：Unity 2021.3 LTS 兼容分支。优先保证稳定、可回归、低依赖。

所有 AI 助手在开始改动前，都应先确认当前分支目标，不要把面向新版本 Unity 的功能直接落到 `unity2021`。

## 分支规则
- 新功能默认先在 `main` 开发。
- 需要支持 Unity 2021.3 的功能，再按需择优移植到 `unity2021`。
- `unity2021` 不接受无明确收益的依赖升级。
- `unity2021` 的改动以兼容修复、API 降级、可选模块拆分、文档补充为主。
- 从 `main` 回移到 `unity2021` 时，优先使用 `cherry-pick`，避免整段合并带入高版本改动。

## 兼容目标
- `unity2021` 的基线版本为 Unity 2021.3 LTS。
- 核心模块必须尽量只依赖 Unity 内置编辑器 API 和 `com.unity.nuget.newtonsoft-json`。
- 涉及 `Timeline`、`Cinemachine`、`Splines`、`ProBuilder`、`UI Toolkit` 等能力时，必须设计为可选模块，不得让核心 asmdef 强依赖这些包。

## 功能演进原则
- 缺失功能可以参考 `Unity-Skills` 的能力边界、命名方式和交互设计。
- 参考不等于直接照搬。优先自己实现，保持 AIBridge 的轻量结构。
- 新命令优先做小而稳定的单职责接口，避免一次性塞入大而全的高耦合命令。
- 涉及版本差异的逻辑，集中收口到兼容层，不要把条件编译散落到整个仓库。

## 代码与结构约束
- 使用 4 空格缩进，保持标准 C# 大括号风格。
- 单个命令类只负责一个明确领域，如 `Scene`、`Prefab`、`Inspector`。
- 新增目录优先遵循现有结构：`Editor/Core`、`Editor/Commands`、`Editor/Utils`、`Tools~`、`Skill~`。
- 尽量保留现有公开命令名，避免无必要的破坏性重命名。

## AI 修改要求
- 修改前先读相关文件，不要凭空假设 Unity API 可用性。
- 新功能若受 Unity 版本限制，必须在提交说明或文档中写明支持边界。
- 修改 `Skill~`、CLI、命令注册逻辑时，要同步考虑文档生成和 AI 发现链路。
- 若新增功能参考自 `Unity-Skills`，在提交说明中注明“参考了能力设计”，但不要大段复制实现。

## 验证要求
- 至少执行一次静态检查，优先使用 `dotnet build` 或对应 CLI 测试。
- 变更编辑器命令时，至少验证命令注册、参数绑定和结果输出链路没有明显回归。
- 涉及 `unity2021` 的改动，优先验证低版本 API 是否可编译，再考虑高版本增强。

## 文档要求
- 分支策略或兼容策略调整时，同步更新 `docs/UNITY2021.md`。
- 新增命令或模块时，补充 README 或 Skill 文档入口说明。
